## 全体アーキテクチャ

> 📝 ここに書くこと: 3つの構成要素（① Python 予測パイプライン → ② JSON スナップショット → ③ Next.js 静的サイト）と、git push で Vercel が自動デプロイされる流れを図で。

```
[国際試合データ] → Python パイプライン（モデル再フィット＋シミュレーション）
                      → web/public/data/*.json
                      → Next.js 静的サイト（output: "export"）
                      → Vercel（main への push で自動デプロイ）
```

- リポジトリ構成: ルートに Python パイプライン、`web/` 配下に Next.js サイト。
- バックエンド不要。パイプラインが書き出した JSON をビルド時に読むだけ。

## 予測モデルの設計

> 📝 ここに書くこと: 「1つのモデルに頼らず複数の見方をブレンドする」という設計思想を最初に述べる。

### Dixon-Coles（過去スコアからの統計モデル）

> 📝 ここに書くこと: 各チームの攻撃力・守備力を過去の試合スコアから推定し、試合の得点をポアソン分布で確率化する古典的手法。ロースコア補正（ρ）と時間減衰（古い試合ほど軽く）にも触れる。

- 中立地開催のためホームアドバンテージは原則ゼロ。開催国（米・加・墨）のみ準ホーム係数を適用。
- 時間減衰パラメータ ξ は過去大会（2022）でチューニング。 // TODO: 確認（ξ の採用値を config.yaml で確認して記載）

### Elo（チームレーティング）

> 📝 ここに書くこと: チェス由来のレーティング。試合結果で増減する相対的な強さの指標。Elo差を勝率・期待得点にマッピングして使う、という説明。

### Shin法でブックメーカーオッズから市場の真の確率を抽出

> 📝 ここに書くこと: ブックメーカーのオッズには控除率（マージン）が乗っている。Shin (1992) 法でそれを除去し、市場が織り込む「素の確率」を取り出す、という話。比例法との違いにも軽く触れると◎。

### 3者ブレンド（重み 0.4 / 0.2 / 0.4）

> 📝 ここに書くこと: チームレベルの優勝確率を `w_dc × P_dc + w_elo × P_elo + w_odds × P_shin` で合成する。なぜ3者を混ぜるのか（それぞれの長所・短所）。

- 重み: Dixon-Coles 0.4 / Elo 0.2 / オッズ(Shin) 0.4。
- 「シム(mix)」は試合レベルで Dixon-Coles と Elo を混合したモデルで、これとは別物。混同しやすいので記事でも整理して書くとよい。

### モンテカルロ・シミュレーション

> 📝 ここに書くこと: 大会の現在状態を初期条件に、残り試合を確率に従って1万回プレーさせ、「優勝した回数 ÷ 1万」を優勝確率とする。乱数シード固定で再現性を担保。

- 既定 10,000 回試行、seed=42。
- グループ順位 → ラウンド32（各組上位2＋成績上位の3位8チーム）→ 決勝の決定論的ブラケット。

## データパイプライン

### 試合結果の取得（martj42 + Wikipedia 補完）

> 📝 ここに書くこと: ベースは martj42/international_results データセット（`fetch-data` で取得・キャッシュ）。大会中の最新結果は手動で補完する運用。

- WC2026 の試合はベースデータから除外し、`data/results_2026.csv` を唯一の正とする（二重カウント防止）。
- データセットが追いつかない直近の結果は Wikipedia の各組ページから補完。 // 例: データセットが6/18で凍結していたため6/19の2試合をWikipediaから追加した、というエピソードを書ける。

### 結果更新 → モデル再フィット → シミュレーション → JSON書き出し

> 📝 ここに書くこと: 毎節後のルーティン。CLI 一連のコマンドを並べて説明。

```bash
python cli.py update-results   # 実結果を取り込み、Dixon-Coles 再フィット＋Elo 更新
python cli.py simulate --model all   # 残り試合を1万回シミュレーション
python cli.py report           # 乖離テーブル＋定性チェック用プロンプト
python cli.py export-web        # web/public/data/*.json を書き出し
```

- 出力 JSON: `latest.json` / `schedule.json` / `matches.json` / `accuracy.json` / `history.json`。

## Next.js 静的サイト

### 静的エクスポート（output: "export"）にした理由

> 📝 ここに書くこと: バックエンドを持たず、ビルド時に JSON を読むだけで完結させたかった。`output: "export"` ＋ `trailingSlash: true` で完全静的化。配信が速く、Vercel との相性も良い。

### 5ページ構成

> 📝 ここに書くこと: 各ページの役割を1行ずつ。

- `/`（ダッシュボード）: 優勝確率トップ15、モデル/オッズ/乖離。
- `/schedule/`（日程・結果）: 全72試合＋決勝トーナメント、順位表。未消化試合はモデル予測。
- `/matches/`（予測検証）: 試合前予測 vs 実結果（的中・Brier）。
- `/timeline/`（確率推移）: スナップショットごとの優勝確率の移り変わり。
- `/accuracy/`（モデル精度）: 累積 Brier・log loss・キャリブレーション。
- （補足）`/about/`（予測のしくみ）: 統計に詳しくない人向けの解説ページも後から追加した。

### キックオフ時刻の JST 化対応

> 📝 ここに書くこと: 元データは日付しか持っておらず時刻情報がなかった。Wikipedia から全104試合のキックオフ時刻を会場ローカル＋UTCオフセットでバックフィルし、UTC で保持して表示時に `Intl.DateTimeFormat({ timeZone: 'Asia/Tokyo' })` で JST 変換した、という話。

- 米・加・墨は複数タイムゾーン（PT/MT/CT/ET）にまたがるので、UTC に正規化してから JST へ変換するのがポイント。
- 時刻は `web/lib/kickoffs.ts` に独立保持（`schedule.json` はパイプラインで再生成されるため）。

## Vercel デプロイの罠

> 📝 ここに書くこと: この記事のいちばんの「ハマりどころ」。リポジトリ直下に Python ファイル（`requirements.txt` 等）があると、Vercel がフレームワークを自動で Python と判定してビルドが失敗する。

- 症状: `Error: No python entrypoint found ...` でデプロイ失敗。
- 原因: ルートの `requirements.txt` により Vercel が Framework Preset を Python と誤検知。
- 回避策: プロジェクト設定の **Framework Preset を "Other"（null）に変更**する。`vercel.json` ではなくプロジェクト設定側で直すのがポイント。
- `vercel.json` には install/build/output を明示:

```json
{
  "installCommand": "cd web && npm install",
  "buildCommand": "cd web && npm run build",
  "outputDirectory": "web/out"
}
```

> 📝 ここに書くこと: 「ビルドコマンドを vercel.json で正しく書いてもフレームワーク誤検知は直らない」点を強調すると、同じ構成（モノレポでルートに別言語）の人に刺さる。

## モデル vs 市場の乖離分析（読み物パート）

> 📝 ここに書くこと: ここがいちばん面白い章。自作モデルと市場（オッズ）の見立てが大きく食い違ったチームを、実際の数値とともに紹介する。乖離 = シム(mix)の優勝確率 − オッズ(Shin)。絶対値3pt以上で⚠。

- **Argentina（強気）**: モデルは高評価だが市場は慎重。 // 例: 6/19時点でモデル 約21.9% に対し市場 約10.7%、乖離 +11.2pt。好スタート（初戦3-0）と高い Elo が効いている。
- **France（弱気）**: 市場が優勝本命に推す一方、モデルは低め。 // 例: 市場 約19.8% に対しモデル 約7.6%、乖離 −12.2pt。対セネガル 3-1 の好発進で市場が反応。
- 実オッズに切り替えたことで Brazil・Spain は乖離が縮小しフラグが外れた、という「市場とモデルが歩み寄った」例も対比として書ける。

> 📝 ここに書くこと: 「モデルと市場どちらが正しいかは大会が答えを出す」という締めにすると読み物として綺麗。数値は記事公開時の最新スナップショットに合わせて更新すること。// TODO: 確認（公開直前に latest.json の最新値で数字を更新）
